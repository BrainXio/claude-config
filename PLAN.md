# Plan: Improve claude-config Pre-Commit Hook

## Context

The current `claude-config/.githooks/pre-commit` has several gaps compared to the project's stated principles and the workspace standards-guard hook:

1. **Assumes local tool installation** â€” checks `command -v mdformat` / `command -v yamllint`. The project principle (in `rules/30-standards/01-markdown-formatting.md` and `AGENTS.md`) is to use `uvx` so no local install is required.
1. **mdformat without plugins** â€” current script runs `mdformat --check` without `mdformat-frontmatter` and `mdformat-gfm`, which destroys YAML frontmatter in agent/rule/skill files.
1. **Attribution guard is narrow** â€” only checks `README.md`, `CONTRIBUTING.md`, `SECURITY.md`. It should cover all staged markdown, YAML, and JSON since these file types contain documentation.
1. **No JSON validation** â€” `settings/` has 11 JSON config files and `workflows/` has JSON workflow files. Invalid JSON in these files breaks the framework.
1. **No standards-guard checks** â€” `claude-standards-guard` (from `claude-cli`) blocks philosophy sludge, manifesto tone, phantom repo links, floating workflow tags, and `curl | bash` patterns. The pre-commit hook does none of this. Since `claude-standards-guard` is a stdin-driven PreToolUse hook, the practical integration is to port its logic into the pre-commit bash script (the workspace `.githooks/pre-commit` already does this).

## Files to Modify

- `claude-config/.githooks/pre-commit` â€” replace with improved version

No other files need changes; `.yamllint` and `.mdformat.toml` configs are already correct.

## Implementation

### 1. Use `uvx` for All Tools

- Markdown: `uvx --with mdformat-frontmatter --with mdformat-gfm mdformat --check <files>`
- YAML: `uvx --from yamllint yamllint -c .yamllint <files>`
- JSON: `uvx --from jq jq empty <file>` (per file, or batch if feasible)

### 2. Broader Attribution Guard

Apply the existing attribution patterns to **all staged** `.md`, `.yaml`, `.yml`, `.json` files, not just three hardcoded ones.

### 3. JSON Validation

For each staged `.json` file, run `uvx --from jq jq empty "$file"`. Fail on invalid JSON with a clear error message naming the file.

### 4. Port `claude-standards-guard` Logic

Add bash checks equivalent to `claude-cli/src/claude_cli/standards_guard.py` for files where it matters:

- **Philosophy sludge** (e.g., `local-first`, `quiet joy`, `sacred`, `sovereign AI`, etc.) â€” check `README.md`, `CONTRIBUTING.md`, `SECURITY.md`, `.github/**/*.md`
- **Manifesto tone** (e.g., `is the only allowed`, `managed exclusively through`) â€” same file set
- **Phantom repo links** â€” check GitHub links in markdown under `.github/` against the `ALLOWED_REPOS` list from `standards_guard.py`
- **Email domain restriction** â€” only `@brainxio.org` permitted in the same guarded files
- **Workflow checks** â€” for `.github/workflows/*.yml` and `*.yaml`:
  - Floating tags: `@latest`, `@stable` in `github.com` URLs
  - Supply chain: `curl | sudo`, `curl | bash`, `curl | sh` patterns

### 5. Script Structure

Keep the same `set -euo pipefail` safety. Structure:

1. Get staged files; exit if none.
1. Markdown formatting check (staged `.md` only).
1. YAML linting (staged `.yaml`/`.yml` only).
1. JSON validation (staged `.json` only).
1. Attribution guard (all staged `.md`/`.yaml`/`.yml`/`.json`).
1. Standards guard (philosophy, manifesto, phantom links, emails, workflow checks).
1. Print pass/fail with actionable error messages.

### 6. Hook Enablement

The script lives in `claude-config/.githooks/pre-commit`. To activate it inside the submodule:

```bash
cd claude-config
git config core.hooksPath .githooks
```

1. **Attribution guard:** Add "Generated by AI" to any `.md` file and stage it. Hook should fail.
